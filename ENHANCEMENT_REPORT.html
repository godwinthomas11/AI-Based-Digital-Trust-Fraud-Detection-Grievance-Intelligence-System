1. Enhancement Requested

As part of the MPJ practical examination, the faculty assigned an enhancement task over and above the baseline project. The specific ask was:

Requirement: “Create a configuration file listing blocked / banned domains. When a scheme message is checked and it contains one of these domains, the system must flag it as suspicious. Additionally, if the same message has been analyzed previously and is present in the analysis database, the system must display the previous score along with a new score computed using the domain-ban configuration.”

The enhancement therefore has two linked parts:

  1. Config-driven domain blocklist — an externally editable file so the list of banned domains can be maintained without touching the Java code.
  2. Historical-aware scoring — if a message is already in the existing analysis table, combine its previous trust score with the penalty from the blocked-domain match so the final displayed score reflects both the older history and the new red flag.

2. My Approach

2.1 Why a .properties file?

A Java Properties file was chosen for the blocklist because it is:

  • Human-readable — any evaluator, faculty, or admin can open and edit it in a text editor.
  • A first-class citizen in Java — loaded in two lines via Properties.load(InputStream).
  • Supports a key-value pair naturally, which fits our need perfectly: domain = penalty_points. This allows different domains to carry different severities.
  • Placed under WEB-INF/, so it is packaged inside the webapp and protected from direct HTTP access.

2.2 Why the combined score uses previous score − penalty

If a message has been seen before, it already carries a trust score in the database. A fresh analysis gives a new score. To satisfy the faculty’s ask of “previous score + new score”, I compute: 

combinedScore = clamp( previousScore − blockedPenalty , 0 , 100 )

This captures both pieces of evidence: the historical trust level and the freshly detected red flag. The result is clamped into the 0–100 range to keep the UI meaningful.

2.3 Minimum-touch, zero-regression design

Existing callers of FraudAnalyzer.analyze(input) continue to work unchanged because the original single-argument method is preserved as a pass-through to the new overload with an empty blocklist. The database schema is also untouched — the existing analysis table is reused for historical lookup.

5.3 AnalyzeSchemeServlet.java

Config load at init()

private Map<String, Integer> blockedDomains = Collections.emptyMap();

@Override
public void init() throws ServletException {
    super.init();
    this.blockedDomains = loadBlockedDomains();
    log("Loaded " + blockedDomains.size() + " blocked domains from config");
}

private Map<String, Integer> loadBlockedDomains() {
    Map<String, Integer> map = new HashMap<>();
    try (InputStream in = getServletContext()
            .getResourceAsStream("/WEB-INF/blocked_domains.properties")) {
        if (in == null) return map;
        Properties props = new Properties();
        props.load(in);
        for (String key : props.stringPropertyNames()) {
            String host = key.trim().toLowerCase(Locale.ROOT);
            if (host.startsWith("www.")) host = host.substring(4);
            try {
                int penalty = Integer.parseInt(props.getProperty(key).trim());
                map.put(host, penalty);
            } catch (NumberFormatException ignored) { }
        }
    } catch (IOException e) {
        log("Failed to read blocked_domains.properties", e);
    }
    return map;
}

6. Deployment Strategy: Two Live Versions

To allow a side-by-side demonstration of the baseline product and the enhanced version, the original deployed webapp was duplicated before any edits were applied. Both copies run under the same Tomcat instance and the same MySQL database, so the “previously analyzed” feature works against a shared history.

VersionURLState
Old (baseline)http://localhost:8080/DigitalTrustProject1_old/Frozen — unchanged copy taken before enhancement
New (enhanced)http://localhost:8080/DigitalTrustProject1/Contains config file, modified Java classes and JSP

Additionally, the Git repository was tagged v1-original-baseline immediately before starting the enhancement work. This provides a second safety net — the full original source tree can be restored at any time with git checkout v1-original-baseline.

8. Demo Walkthrough

  1. Start Tomcat locally using ~/start-trust.sh. Both webapp contexts become available.
  2. Open the old version at /DigitalTrustProject1_old/, log in and analyze a scheme message containing http://fake-scheme.xyz/claim. Note the absence of any “Blocked Domain” row.
  3. Open the new version at /DigitalTrustProject1/, log in, analyze the same message. The result card now shows the new “Blocked Domain” row with the exact penalty and flips the verdict to FRAUD.
  4. Analyze the same message a second time in the new version — “Previously Analyzed” and “Combined Score” rows appear, with the score calculated as previous − penalty.
  5. Open web/WEB-INF/blocked_domains.properties, add a brand-new domain (for example praveetgupta.gov.in=50), copy it to the deployed webapps/ path, restart Tomcat. Analyze a message containing that new domain — it now flags as FRAUD without a single line of Java being touched. This proves the design is truly config-driven.

9. Conclusion

The enhancement delivers exactly what was asked: an externally editable blocklist, runtime enforcement of that blocklist against URLs found in scheme messages, and a combined trust score that fuses historical database evidence with the new domain penalty. The design is deliberately minimal — five files touched, no database schema changes, no breakage of existing callers — which makes the system safe to extend and easy to review.

Outcome: The enhanced build is deployed alongside the original baseline, allowing a direct side-by-side demonstration. A git tag preserves the original source as a second layer of reversibility.
